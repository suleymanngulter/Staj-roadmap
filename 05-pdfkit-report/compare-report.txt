3. PARALELİZASYON ETKİSİ (tüm runtime'larda tutarlı)
================================================================================

Üç ortamda da 8 worker pool ile üretim süresi yaklaşık 3.7–3.8 kat kısaldı.
Bu, Senaryo 3'teki CPU-bound iş + worker_threads modelinin gerçek dünyada
çalıştığını doğruluyor: PDF üretimi ana thread'i bloke eden iş; worker'lara
bölmek çok çekirdekten yararlanıyor.

Paralel kazanç runtime'dan bağımsız ve benzer → darboğaz büyük ölçüde pdfkit
üretim mantığında; runtime seçimi paralel ölçeklenmeyi bozmuyor.

5. GENEL DEĞERLENDİRME
================================================================================

Kazanan (ham üretim hızı)     : Bun
Kazanan (merge / toplam akış) : Node.js (merge hafif avantajı)
En tutarlı paralel ölçekleme  : Node.js (3.79x) — fark küçük
En yavaş alan                 : Deno merge + soğuk başlangıç

Pratik sonuçlar:

  1. CPU-bound PDF üretimi için worker pool üç runtime'da da şart; seri mod
     ~8–10 saniye, paralel ~2.2–2.4 saniye (2000 belge).

  2. Sadece üretim hızına bakılırsa Bun önde. Fark ~800 ms / 2000 PDF (~%8);
     tek seferlik küçük işlerde anlamsız, toplu batch'te birikir.

  3. Üretim + merge birlikte düşünülürse Node.js ve Bun başa baş (toplam ~4 s
     paralel). Deno merge yüzünden ~400–500 ms geride.

  4. Runtime seçimi bu benchmark'ta "hangisi 10x hızlı" sorusunu cevaplamıyor;
     asıl kazanç mimariden geliyor (seri → 8 worker). Runtime farkı %5–10 bandında.

  5. Üretim ortamı önerisi:
     - Mevcut Node ekosistemi / en olgun deploy → Node.js
     - Maksimum üretim throughput, Bun toolchain kabul edilebilir → Bun
     - Deno tercih politikası (izin modeli, tek binary) → performans kabul edilebilir,
       ilk istek/soğuk start ve merge için ek tampon bırak

6. SINIRLAMALAR VE TEKRARLANABİLİRLİK NOTLARI
================================================================================

- Ölçümler aynı makinede farklı saatlerde alındı (Node 11:40, Bun 11:51, Deno 11:54 UTC).
  Arka plan yükü, CPU frekansı ve termal throttling sonuçları etkileyebilir.
- Deno Web Worker, Node/Bun worker_threads'ten farklı API; iş mantığı aynı olsa da
  worker başlatma/mesajlaşma overhead'i küçük fark yaratabilir.
- Son koşudaki merge 2000 PDF birleştirme — ağır I/O + bellek; tek ölçüm noktası.
- Bun koşu 20 seri sıçraması outlier; tekrar ölçümde ortalama düşebilir.

Önerilen tekrar: Üç benchmark'ı arka arkaya, makine boşken, aynı COUNT/WORKERS ile
çalıştırıp bu raporu güncellemek.

7. SONUÇ (tek cümle)
================================================================================

Worker pool olmadan runtime seçimi anlamsız; paralel mimari ~3.7x kazandırır.
